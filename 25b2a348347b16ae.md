# LinkVault Core

LinkVault Core 是一个面向技术团队与内容研究者的结构化外链资源聚合与导航系统。本项目不对资源内容进行二次编辑或篡改，仅提供原始链接的索引、分类、检索与状态监控能力，帮助用户在海量分散的百科类、技术类、文献类页面中快速定位目标条目。项目定位为“链接的链接”，即元索引层，适用于需要批量维护外部参考链接、周期性核查资源有效性、以及按主题组织外部知识图谱的场景。目标用户包括技术文档工程师、数据治理专员、知识库管理员以及进行大规模文献回溯的研究人员。

## 功能概览

**批量链接导入** 支持通过文本文件或标准输入流一次性导入大量 URL，自动去重并生成内部唯一标识。

**自定义分类标签** 允许用户为每个链接附加一个或多个分类标签，支持基于标签的快速筛选与统计。

**链接状态监测** 内置异步 HTTP 健康检查器，可配置超时与重试策略，定期检测每个资源是否可访问，并记录状态变更历史。

**全文元数据提取** 对可访问的链接自动拉取页面标题、描述、关键词及语言编码，用于生成索引卡片。

**结构化导出** 支持将索引数据导出为 JSON、CSV 或 Markdown 表格格式，便于集成到外部文档系统或静态站点生成器。

**查询语法支持** 提供简单的查询表达式，支持按域名、路径关键词、状态码、更新时间范围进行复合条件检索。

**访问统计看板** 记录每个链接的点击次数与最近访问时间，辅助判断资源的热度与老化程度。

## 应用场景

技术文档团队在维护外部参考章节时，可使用 LinkVault Core 统一管理所有引用的百科链接，并定期自动检查链接是否失效，避免文档中出现死链。

数据治理部门在进行外部数据源审计时，可利用本系统导入数百个数据提供方的官方说明页链接，按机构、地区和数据类型进行标签化分类，生成审计报告。

独立研究人员在收集某专题的公开资料时，可以通过本项目快速建立个人参考链接库，并利用元数据提取功能生成初始的文献综述提纲。

知识库运营者可将本系统作为内容中台的上游组件，定期将经过验证的活跃链接同步至生产环境的推荐模块或相关文章列表。

## 快速开始

以下命令演示了从克隆代码到启动服务的完整流程。请确保系统已安装 Git 与 Python 3.10 及以上版本。

```bash
git clone https://github.com/linkvault/linkvault-core.git
cd linkvault-core
python -m venv venv
source venv/bin/activate  # Windows 环境下使用 venv\Scripts\activate
pip install -r requirements.txt
python main.py --init-db --import-links ./samples/links.txt
python main.py --serve --port 8080
```

生产环境部署建议使用 Gunicorn 或 uWSGI 作为 WSGI 服务器，并配合 Nginx 进行反向代理。

## 项目结构

```
linkvault-core/
├── main.py                  # 应用入口，整合 CLI 与 Web 服务启动逻辑
├── requirements.txt         # 生产环境 Python 依赖清单
├── config/                  # 配置模块，支持 YAML 与环境变量覆写
│   ├── default.yaml         # 默认配置，含监听端口、数据库路径、监测间隔
│   └── logging.conf         # 日志轮转与级别配置
├── core/                    # 核心数据模型与业务逻辑
│   ├── models.py            # SQLAlchemy ORM 实体：Link, Tag, StatusHistory
│   ├── validator.py         # URL 规范化、黑名单过滤、协议校验
│   └── indexer.py           # 元数据提取与全文索引构建器
├── checkers/                # 链接状态监测子模块
│   ├── http_checker.py      # 异步 HTTP 探活器，支持重试与代理
│   └── scheduler.py         # 基于 APScheduler 的定时任务调度器
├── api/                     # RESTful API 路由与处理器
│   ├── routes/              # 按资源划分的路由集合
│   │   ├── links.py         # /api/links 的 CRUD 操作
│   │   └── tags.py          # /api/tags 的增删改查
│   └── schemas.py           # Pydantic 请求/响应模型
├── web/                     # 内嵌的简易 Web 看板
│   ├── templates/           # Jinja2 模板文件
│   └── static/              # CSS 与 JavaScript 资源
├── tests/                   # 单元测试与集成测试
│   ├── test_models.py
│   ├── test_checker.py
│   └── fixtures/            # 测试用的模拟链接列表
└── scripts/                 # 运维辅助脚本
    ├── migrate_db.py        # 数据库版本迁移工具
    └── export_csv.py        # 命令行导出器
```

## 贡献指南

1. 查阅 issues 列表，认领未被分配的 bug 或功能需求，或提交新的 issue 描述你遇到的问题或改进建议。
2. 从 main 分支创建新的功能分支，分支命名遵循 feat/xxx 或 fix/xxx 模式，并在分支描述中说明改动范围。

## 常见问题

Q: 系统如何避免对源站点造成过大请求压力？
A: 监测器默认使用 2 秒的请求间隔，且并发数固定为 5。用户可在配置文件中调整 checker.concurrency 和 checker.delay_seconds 参数，建议不超过 10 个并发。

Q: 导入链接时，系统是否会自动过滤掉重复条目？
A: 会。系统以 URL 的标准化形式（去除尾部斜杠、统一协议为小写）作为唯一键，重复导入的链接不会新增记录，但会更新最后发现时间。

Q: 如何迁移数据库到另一台服务器？
A: 直接复制 SQLite 数据库文件（默认为 ./data/links.db）即可。若使用 PostgreSQL 等其他数据库，可使用 scripts/migrate_db.py 配合 --dump 和 --load 参数进行导出导入。
