# WapBaike Resource Aggregator

WapBaike Resource Aggregator 是一个面向技术文档检索、知识库镜像和移动端百科内容聚合的开源工具集。该项目旨在为开发者、研究人员及内容运营者提供一套标准化的外链资源归集方案，能够将分散在不同文章条目中的结构化数据与参考资料进行统一整理、分类索引和本地化存储。

本项目的目标用户包括需要进行大规模百科数据迁移的运维工程师、从事自然语言处理语料构建的数据科学家，以及希望建立私有化移动知识库的产品团队。通过该聚合器，用户可批量获取来自 wap.baike.vokav.cn 的优质文章元数据，并基于这些外链资源构建上层应用，如问答系统、推荐引擎或专题知识图谱。

## 功能概览

**批量文章元数据抓取**：提供基于文章ID的并发请求调度器，支持对指定范围内的百科条目进行标题、摘要、正文关键词及分类标签的自动抽取。

**外链健康状态检测**：内置链接可用性验证模块，定期对已收录的每一篇百科文章中的引用外部链接进行 HTTP 状态码检查，自动标记失效或重定向链接。

**结构化资源索引生成**：将原始的文章 URL 依据内容主题（如科学技术、人文历史、生活健康等）自动归类，生成 Markdown 格式的索引目录，便于人工审阅与二次分发。

**增量更新与去重机制**：支持基于文章最后修改时间的增量拉取策略，避免重复处理未变更资源；同时对文章标题与正文前 200 字符进行 SimHash 计算，剔除内容高度相似的冗余条目。

**多格式数据导出**：支持将聚合后的资源列表导出为 JSON、CSV 以及纯文本 URL 清单三种格式，兼容主流数据库批量导入工具及静态站点生成器的数据源要求。

**可配置的请求限流与重试**：针对移动端百科站点的访问特性，提供可自定义的每秒请求数上限、超时阈值及指数退避重试策略，保障抓取过程稳定且不触发目标服务器的反爬机制。

## 应用场景

**垂直领域知识库初始化**：技术团队在搭建内部 Wiki 或开发者文档门户时，可通过本聚合器快速从海量百科文章中筛选出与编程语言、算法、软件工程相关的条目链接，作为初始语料库的外链基础。

**搜索引擎结果补充**：内容型产品在自身的站内搜索模块中，可将聚合后的百科外链作为补充搜索结果返回给用户，尤其适用于移动端低带宽环境下，以纯文本链接形式提供权威参考资料。

**学术文献引用追溯**：研究人员在撰写综述或进行文献计量分析时，可利用本工具批量导出某类主题下的所有百科文章链接，配合引用网络分析工具，追溯知识传播路径与热点演化趋势。

**日常运维监控看板**：网站运维人员可将本聚合器部署为定时任务，每日生成一份包含文章更新数量、链接失效比例及新增资源类型分布的报告，用于驱动内容运营决策。

## 快速开始

```bash
# 克隆项目仓库至本地
git clone https://github.com/your-org/wapbaike-aggregator.git

# 进入项目根目录并安装依赖（使用 pip 管理 Python 依赖）
cd wapbaike-aggregator
pip install -r requirements.txt

# 运行快速启动脚本，默认从第 29 批次开始抓取，共处理 180 个资源链接
python run_aggregator.py --batch 29 --total 180 --output ./output/
```

## 安装要求

| 依赖 | 必需 | 说明 |
|------|------|------|
| Python 3.8+ | 是 | 核心运行环境，提供异步 I/O 及类型注解支持 |
| requests 2.28+ | 是 | 用于发送 HTTP 请求并处理响应内容 |
| beautifulsoup4 4.12+ | 是 | 解析 HTML 文档结构，提取文章标题、段落及链接 |
| lxml 4.9+ | 是 | 作为 BeautifulSoup 的底层解析器，提升解析效率 |
| aiohttp 3.8+ | 否 | 仅在启用异步并发抓取模式时需要，可显著提高吞吐量 |
| pandas 2.0+ | 否 | 仅在导出 CSV 格式报告时需要，用于数据帧操作 |
| redis-py 4.5+ | 否 | 仅在启用分布式去重和状态共享时需要，用于记录已处理文章 ID |

│   ├── fetcher.py             # 异步请求调度器，含重试与超时控制逻辑

│   └── user_agent.py          # 随机 User-Agent 生成与轮换

## 常见问题

**问：运行聚合器时出现 "Too Many Requests" 或 429 状态码，应如何调整参数？**

答：此响应表明目标服务器正在进行请求限流。请修改 config/settings.yaml 文件中的 rate_limit_per_second 参数，将该值调低至 2 或 3，并适当增大 retry_backoff_factor 至 1.5 以上。同时建议启用 redis 去重功能以避免重复请求同一文章 ID。

**问：部分文章链接返回 404 状态码，聚合器是否会中断运行？**

答：不会。聚合器将 404 视为可接受的非致命错误，会自动记录该链接至 failed_urls.log 文件中并继续处理后续条目。用户可在运行结束后查阅该日志，手动确认文章是否已被删除或 ID 是否有误。

**问：如何将聚合结果导入到本地的 Elasticsearch 或 MySQL 数据库中？**

答：使用 exporter.py 模块的 --format elasticsearch 或 --format mysql 参数即可生成对应的批量导入语句。对于 MySQL，输出为 INSERT 语句的 SQL 文件；对于 Elasticsearch，输出为 ndjson 格式的批量索引请求体。具体用法请参考 docs/operations.md 中的数据库集成章节。
