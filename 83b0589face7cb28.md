# ResourceBridge

ResourceBridge 是一个面向技术研究、内容聚合与知识管理场景的轻量级外链资源汇总系统。该项目旨在为个人开发者、技术内容运营团队以及企业知识管理部门提供一套结构化的外部资源索引与导航方案，帮助用户在信息过载的环境中高效定位、归类与复用有价值的公开技术内容。

与传统的书签管理工具或简单链接收集器不同，ResourceBridge 专注于处理大规模、多批次的资源链接集合，提供基于主题分类、状态标记和来源追踪的组织能力。项目本身不存储资源副本，而是建立高质量的外部信息入口索引，并通过标准化的项目结构便于用户进行二次开发、自动化导入或与其他知识管理工具集成。

项目定位为技术资源导航工具链的基础层组件，适用于需要定期同步、清洗和展示大量外链资源的自动化流程。当前批次为第 19/56 批，共含 180 个资源链接。

## 功能概览

**批量链接标准化入库**：支持从多种数据源（文本列表、CSV、数据库导出）批量导入 URL 列表，自动进行去重、格式校验和协议归一化检查。

**资源状态标记与过滤**：对每条链接标记来源批次、导入时间、有效性状态（可访问性）、内容类型标签，支持按批次、域名、文件扩展名进行过滤检索。

**分类目录自动映射**：依据 URL 路径特征、域名规则或自定义正则表达式，将链接自动归类至预设的技术领域目录（如后端架构、前端工程、运维监控、算法理论等）。

**资源元信息提取**：对可访问的链接尝试提取页面标题、简述和最后修改时间，作为索引字段辅助检索。

**多格式导出接口**：支持将当前批次的资源列表导出为 JSON、YAML 或 Markdown 表格格式，便于嵌入文档或迁移至其他知识库系统。

**变更追踪与差异对比**：记录每次导入批次的资源集合快照，支持对比不同批次间的增减变化，便于审查资源库的演进过程。

## 应用场景

技术博客与资讯周报编辑：技术内容运营人员可定期将一周内收集到的上百篇技术文章链接通过 ResourceBridge 导入，自动生成按主题分类的链接清单，作为周报或月刊的素材底稿，大幅减少手动整理时间。

个人知识库资源同步：个人开发者使用 Obsidian、Notion 或 Logseq 等工具管理学习笔记时，可将各渠道采集的参考链接统一通过本项目整理，再以结构化格式导出至知识库，确保外部参考资料与本地笔记的关联可追溯。

企业技术雷达与外部情报聚合：企业内部架构组或技术委员会可定期爬取或人工提交行业技术大会、白皮书、开源项目发布等链接，通过本项目的批次管理与分类映射能力，构建内部技术趋势观察看板的基础数据层。

开源项目文档的外部引用管理：开源项目维护者在撰写项目文档或用户手册时，需要引用大量外部规范、教程或相关项目地址，可通过 ResourceBridge 维护一个独立的引用链接池，确保文档中所有外部链接均有明确来源、用途和状态标记。

## 快速开始

以下步骤指导你在本地环境快速部署并运行 ResourceBridge 的基础导入与导出流程。

```bash
# 克隆代码仓库
git clone https://github.com/resourcebridge/resourcebridge.git
cd resourcebridge

# 安装项目依赖（使用 pip 和虚拟环境）
python3 -m venv venv
source venv/bin/activate  # Windows 下使用 venv\Scripts\activate
pip install -r requirements.txt

# 执行示例导入：将默认的示例链接列表（位于 data/samples/）导入并生成索引
python bridge.py import --source data/samples/links_19.txt --batch 19

# 导出当前批次为 Markdown 表格
python bridge.py export --batch 19 --format markdown --output output/batch_19_summary.md
```

## 项目结构

```
resourcebridge/
├── bridge.py                    # 命令行入口，注册 import/export/validate 子命令
├── requirements.txt             # 生产环境依赖列表
├── .env.example                 # 环境变量模板（代理、超时、并发等）
├── config/
│   ├── default.yaml             # 默认配置：分类规则、导出格式选项
│   └── custom_rules.yaml        # 用户自定义分类映射规则（正则表达式）
├── core/
│   ├── __init__.py
│   ├── importer.py              # 批量导入逻辑：读取文件、解析URL、去重
│   ├── exporter.py              # 导出模块：支持 JSON / YAML / Markdown
│   ├── classifier.py            # 基于规则和路径特征的自动分类器
│   ├── fetcher.py               # 异步HTTP获取页面元信息（标题、状态）
│   └── diff.py                  # 批次间差异对比（新增、删除、重复）
├── data/
│   ├── batches/                 # 按批次存储原始导入文件与中间结果
│   │   └── batch_19/            # 当前批次目录（导入后生成）
│   │       ├── raw_links.txt    # 原始链接列表副本
│   │       ├── enriched.json    # 补充元信息后的完整条目
│   │       └── summary.md       # 批次汇总报告
```

## 常见问题

问：导入时遇到大量重复链接，如何处理？

答：系统默认在导入过程中基于 URL 完全匹配进行去重。如果你发现同一批次内或跨批次存在大量重复，可以运行 `bridge.py dedupe --batch 19 --strict` 命令进行全局去重检查。对于仅仅是路径大小写或尾部斜杠不同的变体，建议在导入前使用 `normalize` 子命令进行统一格式化。

问：自动分类规则不准确，能否自定义？

答：可以。项目根目录下的 `config/custom_rules.yaml` 文件允许用户定义自己的分类映射规则。规则采用正则表达式匹配 URL 路径或查询参数，并按顺序应用。你可以参考 `config/default.yaml` 中的示例规则格式进行修改。修改后无需重启，下次导入或重新分类时会自动加载。

问：是否可以导入非 HTTP/HTTPS 协议的资源（如 FTP、磁力链接）？

答：当前版本主要针对 HTTP/HTTPS 协议设计，因为这些协议占技术资源链接的绝大多数。对于其他协议，系统会在导入时发出警告并跳过元信息抓取，但会保留原始链接作为记录。如果需要完整支持其他协议，可以通过扩展 `core/fetcher.py` 中的协议处理器来实现。
