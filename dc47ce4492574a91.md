# Vokav Knowledge Bridge

Vokav Knowledge Bridge 是一个面向技术研究者、内容聚合者与知识管理工程师的开源外链资源汇总与结构化导航系统。该项目不对原始内容进行二次编辑或转储，而是以高可读性的目录索引与分类列表形式，将分散于多个子域名下的百科类文章条目进行统一归集与呈现，从而解决跨站点资源检索效率低下、链接散落难以追踪、批次化管理缺失等实际问题。

本项目适用于需要定期批量整理外部引用来源、构建内部知识图谱或对特定域名下的公开内容进行结构化监控的自动化流程。通过提供标准化的资源清单与目录树映射，Vokav Knowledge Bridge 能够与各类爬虫调度器、链接状态检查工具以及文档生成流水线无缝衔接。

## 功能概览

**多源链接归集**：支持同时接入不同子域名下的文章详情页链接，并按原始域名与路径结构自动分组。

**批次化清单管理**：以批次为单位组织资源集合，当前批次编号为 42/56，便于追踪整体采集进度与覆盖率。

**只读式链接呈现**：所有资源均以原始 URL 形式直接列出，不做协议转换、域名补全或路径改写，保证引用溯源的准确性。

**ASCII 目录树映射**：通过文本形式的项目结构示意图，清晰展示资源分类、脚本模块与输出目录之间的层级关系。

**轻量化依赖设计**：基于标准 Python 环境与常用文本处理库，无需额外数据库或消息队列即可完成资源清单的生成与校验。

**表格化文档导航**：提供多维度表格索引，帮助不同角色的使用者快速定位到对应的文档章节与解答方向。

**可扩展的资源字段**：支持在资源列表中添加自定义备注列，用于记录抓取时间、状态码或内容摘要哈希值。

**贡献流程标准化**：内置明确的合并请求检查清单与新增条目标注规范，降低多人协作时的维护成本。

## 应用场景

**技术文档站点外链审计**：在周期性文档审查流程中，使用本项目的资源列表作为待检查 URL 源，配合链接检测工具批量验证各文章详情页的可访问性与内容一致性。

**企业内部知识库索引构建**：将项目输出的结构化链接清单导入企业 Wiki 或 Confluence 页面，作为外部参考资料附录，方便团队成员快速跳转至原始百科文章。

**爬虫任务配置样例库**：以本项目整理后的 URL 集合作为爬虫任务配置文件的数据源，可用于测试爬虫对路径格式变体（如大小写混合的 .sHtML 后缀）的兼容性。

**数据湖原始路径登记表**：在数据湖或对象存储中，将每条链接作为分区键或元数据标签的参考依据，用于标记外部数据来源的原始网络位置。

## 快速开始

以下指令适用于 Linux 与 macOS 开发环境。Windows 用户建议通过 WSL 或 Git Bash 执行。

```bash
# 克隆项目仓库至本地
git clone https://github.com/vokav/knowledge-bridge.git

# 进入项目根目录
cd knowledge-bridge

# 安装所需 Python 依赖（建议使用虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 运行资源列表生成脚本，输出当前批次清单
python scripts/generate_list.py --batch 42 --output ./output/batch_42.md
```

## 贡献指南

1. 新增资源批次时，在 data/batches/ 目录下创建对应的 JSON 文件，文件命名格式为 batch_{批次号}.json，内部使用数组结构存储 URL 字符串。

2. 所有新增或修改的 URL 必须通过 validate_urls.py 脚本的格式检查，包括协议头、域名、路径大小写以及后缀拼写的一致性校验。

## 常见问题

**问：为什么资源列表中的 URL 后缀为 .sHtML 而非标准 .html？**

答：此为源站点的实际路径格式，本项目坚持原样收录原则，不进行任何大小写或后缀改写。任何自动化的链接处理工具应兼容大小写不敏感的文件系统特性，或直接使用本清单中的原始字符串进行访问。

**问：如何确认当前批次的所有链接均有效？**

答：项目提供了 validate_urls.py 脚本，该脚本支持发送 HEAD 请求检查链接可达性，并生成包含状态码与响应时间的校验报告。该功能为可选步骤，默认不强制运行以免对源站点造成额外负载。

**问：能否在资源列表中添加自定义字段，例如分类标签或抓取时间戳？**

答：可以。建议在 data/batches/ 目录下的 JSON 文件中扩展每个条目的对象结构，增加如 "category"、"crawled_at" 等键值对，同时同步更新 schema 文件以保证校验规则的一致性。
