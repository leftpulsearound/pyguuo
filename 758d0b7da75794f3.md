# Knowledge Bridge Directory

Knowledge Bridge Directory 是一个面向技术研究者、开源贡献者和信息检索工程师的轻量化外链资源汇总与导航系统。本项目的核心定位并非构建又一个内容农场，而是通过对高质量外部知识源进行结构化收录、分类索引与可用性监测，为特定垂直领域提供可复用的参考信息枢纽。

项目主要服务于以下三类用户：需要快速查阅技术百科条目与案例数据的开发者；希望从分散的公开页面中提取结构化知识图谱的数据挖掘人员；以及需要在内部文档系统中嵌入可信外部参考链接的团队管理者。区别于通用搜索引擎，Knowledge Bridge Directory 强调链接的长期有效性、分类的可解释性以及索引粒度的精确性，致力于降低从海量信息中定位有效资源的时间成本。

## 功能概览

**批量链接导入与去重**：支持从纯文本列表、CSV 或简易 JSON 清单中批量导入外部 URL，自动执行语法归一化与重复条目剔除，保留原始地址的协议与域名大小写不变。

**多维标签分类体系**：允许用户为每条链接赋予项目、领域、内容类型、语种等多个维度的标签，便于后续按场景筛选和聚合展示。

**可用性健康检查**：内置轻量级 HTTP 状态监测器，可按可配置周期对收录链接进行可达性探测，标记异常状态并生成简易报告。

**结构化元数据提取**：对支持 Open Graph 或 JSON-LD 的页面尝试提取标题、描述与发布日期，辅助生成索引摘要，减少人工编辑负担。

**快速检索与过滤**：提供基于标题关键词、标签组合和收录时间范围的检索接口，支持精确匹配与模糊查询两种模式。

**导出与嵌入能力**：支持将选定的链接列表导出为 Markdown、JSON 或纯文本格式，便于嵌入技术文档、项目 Wiki 或静态站点生成器。

**访问日志统计**：记录链接被点击或查看的次数与时间分布，为资源热度评估提供基础数据支撑。

**协作注释功能**：允许具有编辑权限的团队成员为特定链接添加内部备注、勘误说明或使用心得，提升信息共享效率。

## 应用场景

内部技术文档库的外部参考整合。企业或开源项目的文档体系中往往需要引用大量外部规范、案例或百科条目。Knowledge Bridge Directory 可作为独立的参考链接管理模块，与 MkDocs、VuePress 等静态站点工具配合使用，确保所有外部引用均经过统一审核与定期检查。

垂直领域研究的知识索引构建。从事信息安全、医疗信息化或金融科技等领域的分析师，需要持续跟踪数百个行业术语解释、事故报告或最佳实践页面。本项目提供轻量级索引框架，帮助研究者快速建立领域知识地图，避免重复查找。

自动化测试流程中的外部依赖监控。在 CI/CD 流水线中，可将链接可用性检查作为非阻塞步骤执行，当大量外部链接失效时及时预警，防止产品文档或帮助中心出现大量死链，影响用户体验。

开源社区资源贡献清单管理。社区维护者可使用本项目收录社区成员推荐的优质教程、工具站或学术论文地址，并以清晰的目录结构展示在项目仓库的 README 或 Wiki 中，降低新成员的学习门槛。

## 快速开始

以下步骤帮助您在本地环境中快速启动 Knowledge Bridge Directory 服务实例。

```bash
# 克隆项目仓库至本地
git clone https://github.com/knowledge-bridge/directory.git
cd directory

# 安装项目依赖（使用 pip 与虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 初始化本地数据库并启动开发服务器
python manage.py migrate
python manage.py load_initial_links
python manage.py runserver
```

服务启动后，访问控制台输出的本地地址即可进入链接管理界面。默认管理员账号可通过 `python manage.py createsuperuser` 命令创建。

## 安装要求

| 依赖组件 | 是否必需 | 说明 |
|---|---|---|
| Python 3.9 或更高版本 | 必需 | 核心运行环境，建议使用 3.10+ 以获得更好的类型提示支持 |
| SQLite 3.35+ | 必需 | 默认内置数据库，用于存储链接元数据与标签关系，生产环境可切换至 PostgreSQL |
| Django 4.2 LTS | 必需 | Web 框架及 ORM 层，提供管理后台与请求路由能力 |
| requests 2.31+ | 必需 | 用于执行外部链接的可用性检查与 HTTP 状态获取 |
| beautifulsoup4 4.12+ | 可选 | 启用元数据提取功能时用于解析 HTML 文档结构 |
| celery 5.3+ | 可选 | 若需将健康检查任务异步化或定时化，需安装并配置消息代理 |
| gunicorn 21+ | 可选 | 生产环境推荐的 WSGI 服务器，不适用于开发调试 |
| redis 7.0+ | 可选 | 作为缓存后端或 celery 结果存储，提升高频查询响应速度 |

## 贡献指南

欢迎社区成员参与 Knowledge Bridge Directory 的迭代与完善。请遵循以下步骤提交贡献。

第一，在 GitHub 仓库中创建新的议题（Issue），简要描述您希望添加的新链接分类、功能改进或文档修正建议。核心维护者将在两个工作日内给予初步反馈。

第二，从主分支派生（Fork）个人副本，并在派生仓库中创建功能分支，分支命名遵循 `feature/简述` 或 `fix/简述` 格式。所有代码修改需附带对应的单元测试，确保覆盖率不低于原有水平。

第三，若您提交的是新链接收录建议，请确保每条链接均附带至少一个标签和简要内容摘要。建议使用 `scripts/initial_load.py` 中的格式准备数据，以便通过自动化校验。

第四，提交拉取请求（Pull Request）前，请执行 `python manage.py test` 确保全部测试通过，并运行 `flake8` 或 `black` 进行代码风格检查。拉取请求描述中需关联相关议题编号。

第五，拉取请求合并后，您的用户名将被自动记录在 `CONTRIBUTORS.md` 文件中。对于持续贡献者，将酌情授予直接推送权限。

## 常见问题

**问：收录的链接如果失效了怎么办？系统会自动处理吗？**

答：系统内置的健康检查服务会按照可配置周期（默认每 72 小时）对所有链接发起 HEAD 或 GET 请求并记录状态码。连续三次检查均返回 4xx 或 5xx 状态码的链接将被标记为“异常”，并在管理后台的仪表板中高亮显示。项目维护者会定期审查异常链接并尝试更新地址或移除条目，但系统不会自动删除任何链接，以避免误判导致信息丢失。

**问：我可以将本项目作为子模块嵌入到现有的 Django 项目中吗？**

答：可以。Knowledge Bridge Directory 的应用模块（apps/links 与 apps/checks）均设计为可插拔组件。您只需将 `apps/` 目录复制到目标项目，在 `INSTALLED_APPS` 中注册，并运行迁移命令即可。但请注意，本项目依赖的模板和静态文件路径可能需要根据宿主项目的目录结构进行微调。详细集成步骤请参考 `docs/developer-guide/integration.md`。
